How to use the cards

Read & Refine

If you're working on a predefined task

Read the task carefully and understand what it is asking you to do.

Examples

  • The README suggests this should be a command line application, so no website required.
  • The task guidance asks me to always present numbers in the form £0.00, so I will make sure to do that.

If you're working on your own idea

Refine the idea down into something you can build.

Examples

  • I wanted to build a web app of some kind to improve my web development skills. After thinking about it, I decided to build a notes app.
  • I wanted to do something fun and interesting. After throwing some ideas around, I decided to build a web-based PacMan clone.

Start a timer for 15 minutes

If it goes off before you complete Implement, jump to Revert

Plan

Work out what you're going to do, and do any planning necessary.

Keep it small

You will need to plan, write the test, and make it pass in 15 minutes or less.

Think from the user’s perspective

If possible, your task should be from the user's perspective, for example:

Good

The user makes a deposit and then sees it on their bank statement.

Bad

I add a class with a deposit method and a print statement method.

Use diagrams to clarify your thinking

This is a good time to draw some diagrams.You might want to figure out the objects that you will use to break up your solution, or a process diagram to get the sequence straight in your head.

Test

Write a test for the piece of functionality you are about to add.

Test from the user’s perspective first

Think about how the user will use it and write the test from that perspective. Don't create any classes or do any implementation work until you’ve done that.

Example

it "prints a statement with one deposit" do
  bank = Bank.new
  bank.deposit(1000)
  expect(bank.print_statement).to eq "1000"
end

Implement

Make the test pass.

Your whole test suite should be green by the end of this step.

Do the simplest thing that could possibly work

Don't think too far ahead.

Hard-coding it is acceptable — you can plan more versatile behaviour in the next planning step.

Refactor

Improve the code without changing what it does.

Try these strategies

  • Running a linter and resolving all the warnings.
  • Look for large methods or classes and break them up.
  • Check your test descriptions for clarity and correctness.
  • List out the responsibilities of each of your classes.

Reflection

Look through your code and make a list of everything you like about it.Then look through again and make a list of everything you don’t like.

Improve the things that you can right now. Everything else, make a note of as an area to read up on or ask someone about.

Commit

Commit your changes.

Write a descriptive subject line

Phrase your message as if you were explaining briefly what you’ve been doing since the last commit.

Good

Extract a Weather class out of Airport

Bad

Did some general refactoring

Reflection

If you add a blank line beneath the commit subject you can add a description.You can use this area to give more context or to reflect upon what you did and any thoughts you might have about it.

Example

Extract a Weather class out of Airport

I realised Airport had responsibility for landing
planes and determining the current weather. I thought
this violated the Single Responsibility Principle so I
created a new class and some unit tests. Now Airport is
smaller and more focused.

Now head back to plan and do another cycle

The cards below are for if your timer goes off.

Revert

Revert to the last commit that passes the tests.

You can do this with git add . && git stash The 15 minute rule

The 15 minute rule

You should play this card if it has been more than 15 minutes since you started planning.

This is the most important card in the whole deck.

By halting after 15 minutes and trying something simpler, you avoid going off on tangents or picking tasks tasks too big to tackle at once.

It also helps you refine your process and reflect. Each time you revert you will notice things you do slightly differently. Over time, you can keep the things that work, and drop the things that don’t.

Simplify

Take your previous plan and make it simpler.

Try these strategies

  • Doing fewer things at once.
    E.g. if you’re trying to set up a web server with a database, try just setting up the web server.
  • Solving a smaller version of the problem.
    E.g. if you’re making a 10x10 game board, try making a 2x2 one.
  • Hard-coding something
    E.g. if you’re trying to send an SMS with the current time in it, try sending the SMS with a fixed time.

You can re-add the complexity in a future iteration.
